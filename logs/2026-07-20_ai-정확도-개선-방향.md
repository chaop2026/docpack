# SafeFile AI 정확도 개선 방향 (2026-07-20)

SafeFile PII 탐지의 오탐(false positive)·미탐(false negative)을 줄이기 위한 방향을 정리한다.
개인정보 원문을 **절대 수집·커밋하지 않는다**는 프라이버시 원칙(전 과정 브라우저 내 처리)을 모든 항목의 상위 제약으로 둔다.

## 배경

- 탐지 리다크션 메커니즘(좌표 오버레이·플래튼·0 텍스트 누출)은 안정화됨. 남은 결함은 대부분 **탐지 recall/precision**(무엇을 찾는가)이지 리다크션(어떻게 가리는가)이 아님.
- 골든 하니스: `test/safe/run.mjs` + `test/safe/specs.json`. 픽스처는 `generate_fixtures.py`로 **합성 생성**(원문 미포함, `testdocs/`는 gitignore). 실물 결함은 합성 픽스처로 재현해 회귀 테스트로 고정하는 패턴 확립됨(예: 2줄 이름 래핑 → `9_resume_2line.pdf`).
- 알려진 약점: 이름·회사명은 서버 AI 정밀검사 의존, 한국어 저화질 스캔 OCR recall 미흡, 주소 정규식 span 근사.

## 1. 실물 문서 테스트셋 구축 (30 → 100건)

**목표**: 합성 픽스처만으로는 못 잡는 실제 문서 다양성(레이아웃·폰트·언어·스캔품질)을 골든에 반영해 회귀 커버리지 확대.

**핵심 제약 — 원문 커밋 금지, specs만 커밋**
- 실물 문서(이력서·증명서 등) 원본은 **repo에 커밋하지 않는다.** `testdocs/`는 계속 gitignore.
- 커밋 대상은 **specs 항목뿐**: 각 문서에서 "무엇이 마스킹돼야/안 돼야 하는가"를 기술한 기대값(`must_mask`/`must_not_mask`/`should_candidate`/`never_candidate`/…).
- 실물에서 결함을 발견하면 → 그 결함 패턴을 **합성 픽스처로 재현**(`generate_fixtures.py`) → specs에 회귀 케이스 추가. 실물 자체는 로컬 검증에만 쓰고 버림.
- 즉 "실물로 발견 → 합성으로 고정"의 파이프라인을 30건 → 100건 규모로 확장.

**작업 항목(초안)**
- [ ] 문서 유형 매트릭스 정의(이력서/증명서/급여명세/계약서 × ko/en/ja/es × 텍스트PDF/스캔/사진).
- [ ] 실물 로컬 검증 러너: 임의 로컬 문서를 넣고 탐지 결과를 사람이 라벨링 → specs 초안 자동 생성(원문은 저장 안 함).
- [ ] 발견 결함마다 합성 재현 픽스처 + specs 회귀 케이스 추가(현재 130 → 목표 규모 확대).

## 2. 결과 화면 오탐/미탐 신고 버튼 (원문 미수집 설계)

**목표**: 사용자가 결과 화면에서 "이건 가렸어야 했다(미탐)" / "이건 가리면 안 됐다(오탐)"를 신고 → 탐지 개선 피드백 루프 확보.

**핵심 제약 — 원문 미수집 설계**
- 신고 payload에 **문서 원문·PII 값 자체를 절대 담지 않는다.**
- 대신 **비식별 신호**만 수집: 엔티티 `type`, `kind`(auto/candidate/manual), `reason`, 값의 **형태 특징**(길이·문자클래스 패턴·토큰 수 등, 원문 복원 불가한 형태), 문서 유형, 언어, 페이지 상 대략 위치. 실제 문자열은 전송 금지.
- 서버 저장도 비식별 신호만. 신고는 익명·집계 목적.

**작업 항목(초안)**
- [ ] 신고 UI: 각 엔티티 행 / 결과 뷰어에 "오탐"·"미탐" 신고 액션. 미탐은 사용자가 놓친 영역을 지정(위치만).
- [ ] payload 스키마 설계 + "원문 미포함" 자체 검증(전송 전 assert).
- [ ] 수집 엔드포인트(집계 저장) + 관리자에서 신고 패턴 열람 → 1번 테스트셋/프롬프트 개선에 환류.

## 3. 전용 모델 파인튜닝 — 검토 후 기각

**결정**: SafeFile 전용 PII 탐지 모델 파인튜닝은 **기각**.

**대체 방향**: **프롬프트 · 퓨샷(few-shot) · 피드백 루프**로 정확도를 끌어올린다.
- 프롬프트 엔지니어링: 오탐/미탐 패턴을 시스템 프롬프트 규칙으로 반영.
- 퓨샷: 대표 결함 케이스를 예시로 주입(비식별/합성 예시).
- 피드백 루프: 2번 신고 데이터(비식별) → 프롬프트·퓨샷·specs 개선 반복.

**기각 사유(요지)**: 파인튜닝은 학습 데이터에 실물 PII가 필요해 프라이버시 원칙과 충돌 위험이 크고, 운영·비용·업데이트 부담 대비 프롬프트/퓨샷/피드백 루프로 얻는 개선 폭이 충분하다고 판단.
